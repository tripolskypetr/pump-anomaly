# План: математически доказуемое обучение, а не брутфорс

## Проблема, которую решаем

Текущий `fit` на `DEFAULT_GRID` перебирает 279 936 конфигураций и берёт лучшую по CV.
При малой выборке это **гарантированный data-snooping**: максимум из N шумных оценок
смещён вверх на ≈ σ·√(2·ln N) даже когда истинный edge = 0. С N=279936 это смещение
≈ 5σ — то есть система «найдёт» эдж в чистом шуме почти наверняка.

1-SE правило и shrinkage **уменьшают** смещение, но не дают **доказательства**, что
выбранная стратегия — не выброс. Для гос-банка нужен не «score повыше», а
**статистический сертификат**: вероятность, что эдж — артефакт перебора, ниже порога.

## Что отличает edge от выброса (математический аппарат)

Добавляем четыре независимых инструмента, каждый атакует свой источник ложного эджа:

### 1. Deflated Sharpe Ratio (DSR) — López de Prado 2014
Корректирует наблюдаемый Sharpe на (а) число испытаний N, (б) ненормальность
(skew/kurtosis), (в) длину ряда. Возвращает **вероятность, что истинный SR > 0**
с учётом того, что мы выбрали лучший из N.

- Сначала считаем «ожидаемый максимальный SR при истинном эдже 0»:
  `SR0 = √V · [(1−γ)·Φ⁻¹(1−1/N) + γ·Φ⁻¹(1−1/(N·e))]`, где V — дисперсия SR по
  испытаниям, γ — Euler-Mascheroni, Φ⁻¹ — обратная нормаль.
- DSR = `Φ( (SR_observed − SR0)·√(T−1) / √(1 − skew·SR + (kurt−1)/4·SR²) )`.
- **Решение:** стратегия проходит, только если DSR ≥ 0.95 (95% уверенности, что
  эдж реален с поправкой на N испытаний). На 17 сделках и N=280k DSR будет ≈ 0 —
  система ЧЕСТНО скажет «не доказано», вместо ложного «reliable».

### 2. Probability of Backtest Overfitting (PBO) — Bailey & López de Prado 2015
Combinatorially-Symmetric Cross-Validation (CSCV): матрицу результатов
(конфиг × фолд) делим на S частей, перебираем все C(S, S/2) разбиений на IS/OOS,
на каждом выбираем лучший по IS и смотрим его ранг на OOS. PBO = доля случаев,
где IS-лучший попал в **нижнюю половину** OOS. PBO → 0.5 означает чистый оверфит.

- **Решение:** PBO ≤ 0.1 (best IS-конфиг остаётся хорошим OOS в ≥90% разбиений).

### 3. White's Reality Check / Hansen SPA — через stationary bootstrap (Politis-Romano)
Тест на data-snooping: H0 = «лучшая из N стратегий не лучше бенчмарка (0)».
Stationary bootstrap (блочный, сохраняет автокорреляцию ряда сделок) генерирует
B ресэмплов, на каждом считаем макс-статистику; p-value = доля бутстрэп-максимумов,
превысивших наблюдаемый. Блочность обязательна — сделки во времени зависимы.

- **Решение:** p-value ≤ 0.05.

### 4. Минимальная длина выборки (minTRL) — López de Prado
Сколько сделок НУЖНО, чтобы наблюдаемый SR был значим на заданном уровне:
`minTRL = 1 + [1 − skew·SR + (kurt−1)/4·SR²]·(Z_α / SR)²`.
Если фактическое N < minTRL — выборки физически не хватает, любой вывод
преждевременен. Это даёт тебе **число сделок до доверия**, а не «обучилось/нет».

## Принцип отбора конфигурации (заменяет argmax)

Конфиг допускается в прод ТОЛЬКО если выполнены ВСЕ:
1. DSR ≥ 0.95 (эдж переживает поправку на N испытаний),
2. PBO ≤ 0.10 (не оверфит по CSCV),
3. SPA p-value ≤ 0.05 (не data-snooping),
4. N ≥ minTRL (выборки достаточно),
5. nested-CV OOS-score > 0 (несмещённый прод-прогноз положителен).

Если хоть одно не выполнено → `certified=false`, модель торговать НЕ должна.
Это и есть «математически доказуемо»: не порог score, а четыре независимых
статистических барьера против разных механизмов ложного эджа.

## Сжатие грида (борьба с N испытаний у источника)

Брутфорс 280k — главный враг DSR. Снижаем N испытаний структурно:
- **Coordinate descent** вместо полного декартова произведения: оптимизируем оси
  по очереди (exit → cascade → detector), фиксируя остальные. ~10·(сумма длин осей)
  испытаний вместо произведения. Для дефолтного грида ≈ 200 вместо 280k.
- **Purged K-Fold** (López de Prado): удаляем из train сделки, перекрывающиеся по
  времени с val (purge) + эмбарго после val, чтобы автокорреляция не текла между
  фолдами. Иначе CV-оценка оптимистична.

Меньше N → DSR-порог достижим при меньшем числе сделок. Coordinate descent с N≈200
требует ~10× меньше данных для той же доказуемости, чем брутфорс N=280k.

## E2E валидация (test/e2e/*)

Генерируем синтетические последовательности по 500 сигналов с ИЗВЕСТНОЙ истиной:
- **edge-present**: заложен реальный эдж (сигналы предсказывают рост) → аппарат
  ДОЛЖЕН сертифицировать (DSR высокий, PBO низкий, p-value мал, N≥minTRL).
- **edge-absent (чистый шум)**: сигналы независимы от будущего → аппарат ДОЛЖЕН
  отказать (DSR≈0, PBO≈0.5, p-value>0.05). Это главный тест: ловит ли система
  выброс-в-шуме, который брутфорс бы «сертифицировал».
- **edge-with-outlier**: околонулевой эдж + один гигантский выброс, дающий
  положительный mean. DSR/SPA должны НЕ сертифицировать (эдж держится на 1 точке).
- **regime-shift**: эдж в первой половине, шум во второй → purged-CV и nested OOS
  должны поймать распад.

Каждый прогон — 500 сигналов, чтобы N был достаточен и тесты различали кейсы.

## МЕТА-winner's-curse: размножение испытаний через частый refit (Tripolsky)

DSR штрафует N конфигов ВНУТРИ одного fit. Но скользящее переобучение (например,
ежечасно → ~720 fit/месяц) и торговля только при certified=true — это повторный
перебор УЖЕ ПО ВРЕМЕНИ. Каждый «сертифицированный» fit может быть выбросом среди
720 попыток. Сертификат на одном fit слеп к цепочке fit-ов.

Лечение (двухчастное, src/meta-ledger.ts):

1. **CADENCE GUARD** (`canRefit`): fit разрешён не чаще `minRefitMs` (по умолчанию
   неделя). Частые refit размножают испытания → запрещаем на уровне политики.

2. **FAMILY-WISE коррекция** (`effectiveTrials`): nTrials для DSR = Σ конфигов по
   ВСЕМ fit-попыткам (`metaLedger`), а не только текущему гриду. ВСЕ попытки
   логируются — и certified, и нет — иначе знаменатель занижен и поправка лжёт.

Доказано (test/meta-ledger.test.ts): 720 fit на коротком шуме → наивный DSR ложно
сертифицирует часть, family-wise DSR (N×720) — 0. При этом СИЛЬНЫЙ эдж (0.75σ,
500 сделок) переживает даже 720 fit — поправка топит только пограничные/ложные
эджи, зависящие от мета-перебора, а не реальный сигнал.

Применение: храни `MetaLedgerState` в памяти между прогонами, передавай в
`fit(history, gc, { metaLedger })`, проверяй `canRefit` перед каждым fit. Модель
несёт `effectiveTrials` и `fitAttempts` для аудита.
